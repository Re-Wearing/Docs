**1.  Introduction**

본 문서는 우리 팀에서 개발하고자 하는 웹사이트인 'RE:WEAR'의 Software
Design Specification(SDS)이다.

RE:WEAR는 계절이 바뀌거나 유행이 지난 옷들이 옷장 속에 방치되는 문제를
해결하고, 동시에 사회적으로 도움이 필요한 곳에 도움을 제공하는 것을
목표로 한다. 본 문서에서는 RE:WEAR의 기능적 요구사항을 기반으로 시스템을
기능적 관점에서 분석하고 설계하였다.

Use case diagram은 사용자의 관점에서 웹사이트가 제공하는 주요 기능(기부
신청, 직접 매칭, 자동 매칭, 물품 게시판, 배송 상태 조회 등)을 표현하며,
Class diagram은 웹사이트를 구성하는 객체 간의 관계를 나타낸다. 또한
Sequence diagram과 State machine diagram은 시스템의 동작을 묘사하고,
User interface 항목에서는 사용자가 직접 접하게 되는 화면 요소와 상호작용
방식을 설명한다.

본 문서 작성 시 가장 중점적으로 고려한 부분은 Diagram 간의 일관성이다.
모든 Diagram이 SRS 문서에서 제시된 기능 요구사항을 충실히 반영하도록
하였으며, SRS와 SDS 간의 모순이나 누락이 발생하지 않도록 유의하였다.

RE:WEAR는 기부자, 기관, 관리자 세 유형의 사용자가 상호작용하는 구조로
설계되어 있으며, 사용자는 웹사이트를 통해 손쉽게 기부를 신청하고 진행
상황을 확인할 수 있다. 본 문서에서는 이러한 기능들을 중심으로 시스템
구조를 나타내며, 개발 및 유지보수 단계에서 일관된 참조가 가능하도록
하였다.

**4. Sequence diagram**

4.1 이메일 인증코드 발송

![](image/media/image9.png)

\<그림 4-1\>

위 \<그림 4-1\>은 사용자가 이메일 인증을 위해 인증코드를 요청하는 과정을
나타낸 시퀀스 다이어그램이다.

사용자는 이메일 주소를 입력하여 AuthController에 전송하고, 컨트롤러는
AuthService의 sendVerificationCode(email)을 호출한다. 서비스는 기존
미사용/만료 레코드를 정리한 뒤 토큰과 만료시각을 생성하여
EmailVerificationRepository에 저장한다. 이어 EmailService를 통해
사용자에게 인증코드가 포함된 메일을 발송하고, 성공 여부를 확인한 후
컨트롤러로 결과를 반환한다. 컨트롤러는 200 OK로 응답하며, 실패 시에는
메일 발송 오류 또는 저장 실패에 대한 예외를 적절한 상태코드로 변환해
반환한다. 이 흐름은 발급·저장·발송의 순차 단계와 오류 발생 시 alt 분기를
통해 사용자에게 즉시 피드백을 제공한다는 점이 핵심이다.

4.2 이메일 인증코드 검증

![](image/media/image10.png)

\<그림 4-2\>

위 \<그림 4-2\>는 사용자가 받은 인증코드를 제출하여 이메일을 검증하는
과정을 나타낸다.

사용자가 email과 code를 제출하면 AuthController는
AuthService.verifyCode(email, code)를 호출한다. 서비스는
EmailVerificationRepository에서 해당 토큰을 조회하고, 존재 여부·만료
여부·사용 여부를 검증한다. 다이어그램의 alt 블록은 토큰이 없거나
만료·이미 사용된 경우를 실패 분기로 표시하며, 이때 컨트롤러는 400 Bad
Request 등으로 응답한다. 유효한 경우 서비스는 토큰을 사용 처리하고
컨트롤러로 성공을 반환한다. 컨트롤러는 세션/컨텍스트에
emailVerified=true와 같은 플래그를 설정한 뒤 200 OK로 응답한다.
요약하면, 본 흐름은 "조회→검증→사용처리→세션표시"의 순서로 진행되며,
모든 실패 케이스는 alt로 명확히 분기된다.

4.3 로그인

![](image/media/image11.png)
\<그림 4-3\>

위 \<그림 4-3\>은 사용자가 아이디/비밀번호로 로그인하는 과정을 표현한다.

사용자가 로그인 요청을 보내면 UserController는
UserService.login(username, password)를 호출한다. 서비스는
UserRepository에서 사용자를 조회하고(user not found 분기), 존재할 경우
암호화 모듈(예: BCryptPasswordEncoder)을 이용해 비밀번호 일치 여부를
검증한다. 비일치 시 alt 분기를 통해 인증 실패로 종료되며 컨트롤러는 401
Unauthorized를 반환한다. 일치하는 경우에는 사용자 엔티티/DTO를 반환하고,
컨트롤러는 세션 또는 토큰 발급(필요 시)을 수행한 뒤 200 OK로 응답한다.
본 다이어그램은 사용자 존재 유무와 자격 증명 일치 여부의 두 단계 검증을
중심으로 성공/실패 흐름을 명확히 구분한다.

4.4 기부 등록

![](image/media/image12.png)

\<그림 4-4\>

위 \<그림 4-4\>는 사용자가 기부를 등록하고(필요 시) 배송을 자동 생성하는
흐름을 나타낸다.

사용자가 기부 정보를 제출하면

DonationController는 DonationService.registerDonation(\...)을 호출한다.
서비스는 요청 데이터를 검증한 뒤 DonationRepository에 상태 REQUESTED로
저장한다. 시스템 정책에 따라 자동 배송 생성이 활성화된 경우(다이어그램의
opt 블록), 서비스는 DeliveryService를 호출하여 DeliveryRepository에 신규
배송을 CREATED 상태로 기록하고 추적번호 등을 부여한다. 모든 처리가
완료되면 컨트롤러는 생성된 자원 식별자와 상태를 포함하여 201 Created로
응답한다. 검증 실패나 저장 오류 시에는 해당 분기에서 예외를 발생시켜
적절한 오류 응답으로 변환한다. 핵심은 "등록→상태부여→(선택)배송생성"의
단계적 처리와 선택 흐름을 opt로 분리한 점이다.

4.5 관리자 기관 승인

![](image/media/image13.png)

\<그림 4-5\>

위 \<그림 4-5\>는 관리자가 대기(PENDING) 상태의 기관을 승인하는 절차를
나타낸다.

관리자가 승인 요청을 보내면

AdminOrgController는 OrganService.approve(organId)를 호출한다. 서비스는
OrganRepository에서 기관을 조회하고, 존재하지 않을 경우 alt not found
분기로 컨트롤러에 오류를 반환한다. 기관이 존재하며 상태가 승인 가능할
경우, 서비스는 상태를 APPROVED로 갱신하고 저장한다. 이후 컨트롤러는 성공
메시지와 함께 관리자 화면(예: 대기 목록)으로 리다이렉트하거나 200 OK를
반환한다. 본 다이어그램은 조회→검증→상태변경의 관리 절차를 단순화해
표현하며, 존재하지 않음/상태 부적합 등 예외 케이스를 alt로 처리하여
운영자 피드백을 명확히 한다.

**5. State machine diagram**

5.1 이메일 인증코드 상태

![](image/media/image14.png)

\<그림 5-1\>

위 \<그림 5-1\>은 이메일 인증에 사용되는 인증코드 토큰의 생애주기를 상태
기계로 표현한다. 초기 상태에서 토큰은 Created로 생성되며, 메일 발송이
성공하면 Sent로 전이된다. 사용자가 올바른 코드를 제출하면
유효성(만료·사용 여부 포함) 검증을 거쳐 Verified로 확정된다. 만료 시각을
경과할 경우 Expired로 이동하며, 관리자가 강제 취소하거나 사용자가 요청을
철회하면 Revoked로 전이된다. 재발송의 경우 동일 상태 내에서 코드 갱신과
만료 연장 액션이 수행된다. 본 상태도는 발급→발송→검증→종결의 순환을
명확히 정의하여, 동일 이메일에 대한 중복 발급·중복 사용 방지와 만료 정책
준수를 보장한다.

5.2 사용자 이메일 검증

![](image/media/image15.png)

\<그림 5-2\>

위 \<그림 5-2\>는 사용자의 이메일 검증 절차를 사용자 관점에서 나타낸다.
초기 Unverified에서 사용자가 코드를 요청하면 CodeRequested를 거쳐
WaitingCode로 진입하며, 실제 메일 발송 완료 시점과 사용자 입력 대기
상태를 구분하였다. 사용자가 코드를 제출하면 유효한 경우 Verified로
확정되고, 잘못된 코드·만료·오입력의 경우에는 Unverified로 되돌아가 오류
메시지를 제공한다. 쿨다운 조건을 충족하면 resend()를 통해 재요청 경로가
열린다. 이 흐름은 UI/UX에서 필요한 버튼 활성화 조건(재요청, 취소)과 서버
측 검증 타이밍(발송 성공 이후 입력 허용)을 동기화하기 위한 근거로
사용된다.

5.3 로그인/세션 상태

![](image/media/image16.png)

\<그림 5-3\>

위 \<그림 5-3\>은 계정 인증과 세션 관리를 위한 상태 전이를 정의한다.
LoggedOut 상태에서 자격 증명 제출 시 Checking으로 이동하여 사용자 존재
여부와 비밀번호 일치 여부를 검증한다. 일치하면 Authenticated로 전이되어
세션 또는 토큰이 발급되고, 불일치 또는 미존재의 경우 실패 카운트를
증가시키며 LoggedOut으로 되돌린다. 실패 횟수가 정책 임계치에 도달하면
Locked로 전환되어 일정 시간 또는 관리자 해제 전까지 로그인이 차단된다.
인증된 상태에서는 명시적 로그아웃, 토큰 만료, 혹은 갱신 성공 여부에 따라
각각 LoggedOut으로 복귀하거나 상태를 유지한다. 본 정의는 보안
정책(락아웃, 토큰 만료/갱신)과 사용자 경험(실패 메시지, 재시도 시점)을
일관되게 보장한다.

5.4 기부 상태

![](image/media/image17.png)

\<그림 5-4\>

위 \<그림 5-4\>는 기부 건의 처리 단계를 상태로 모델링한다. 사용자가
등록하면 Requested로 시작하며, 내부 심사 또는 매칭 로직에 의해
Matched(수혜기관 배정)로 전이되거나, 심사 실패 시 Rejected, 사용자가
철회하면 Cancelled로 종료된다. 매칭 이후에는 배송
준비(PreparingShipment)를 거쳐 택배 인계 시 Shipped, 배송사 확인 완료 시
Delivered, 수령자 확인 또는 자동완료 정책에 따라 Completed로 종결된다.
배송 중 반송 프로세스가 개시되면 Returned를 통해 Cancelled로 귀결된다.
이 상태도는 각 단계별 허용 작업(수정 가능 여부, 취소 가능 여부)과
대시보드 집계 기준(요청·매칭·완료율)을 결정하는 근거로 사용된다.

5.5 배송 상태

![](image/media/image18.png)

\<그림 5-5\>

위 \<그림 5-5\>는 배송 단위의 세부 상태를 정의한다. 생성 시 Created에서
시작하여 집하 시 PickedUp, 허브 이동 중 InTransit, 최종 배달 전
OutForDelivery, 수취 증빙 확보 시 Delivered로 전이되며, 보관·종결 처리를
통해 Closed로 마감된다. 예외적으로 허브 보류(Delayed), 배달
실패(FailedAttempt), 반송(ReturnToSender → Returned), 그리고
분실·파손(Lost/Damaged) 경로가 정의되어 있다. 재시도 가능 조건과
타임아웃에 따른 자동 반송 로직을 명확히 하여, 실시간 상태 표시, 알림
트리거, 정산 기준(분실·파손 보상 등)을 체계화한다.

5.6 기관 승인 상태

![](image/media/image19.png)

\<그림 5-6\>

위 \<그림 5-6\>은 기관 등록과 승인 이후 운영 상태를 다룬다. 신규 등록은
Pending으로 분류되며, 관리자 검토 결과에 따라 Approved 또는 Rejected로
전이된다. 승인 이후 정책 위반이나 운영 중지 사유 발생 시 Suspended 혹은
Deactivated로 전환되며, 필요시 reinstate()를 통해 Approved로 복귀한다.
Rejected나 Deactivated 상태는 정리 정책에 따라 Deleted로 영구 삭제될 수
있다. 이 모델은 관리자 화면의 버튼 활성화(승인·거절·중지·복구)와 API
권한 제어(승인 전 데이터 비공개, 중지 시 신규 요청 차단)를 일관되게
적용하기 위한 기준을 제공한다.

**8. Glossary**

+--------------+-------------------------------------------------------+
| 이름         | 설명                                                  |
+--------------+-------------------------------------------------------+
| 기관(Organ)  | 일반 회원이 기부한 물품을 받는 복지기관이나 단체인    |
|              | 회원 유형이다.                                        |
+--------------+-------------------------------------------------------+
| 관           | 기관 계정의 회원가입을 승인 및 거절, 공지나 FAQ를     |
| 리자(Admin)  | 관리하고 이상행위에 대해 대응하는 등의 총관리를       |
|              | 맡는다.                                               |
+--------------+-------------------------------------------------------+
| 기           | 일반 회원이 자신의 물품을 나누기 위해 진행하는 과정을 |
| 부(Donation) | 말한다. 물품을 등록하고 해당 물품을 신청하는 단계를   |
|              | 통해 기부가 이뤄진다.                                 |
+--------------+-------------------------------------------------------+
| 물품(Item)   | 일반 회원이 나누고자 하는 물품 자체를 말한다. 해당    |
|              | 물품에 대한 종류, 상세 설명, 물품 이미지를 하나의     |
|              | 객체로 묶어 프로젝트에서 사용한다.                    |
+--------------+-------------------------------------------------------+
| 매           | 기부 신청한 물품과, 수령 가능한 기관을 연결하는 절차. |
| 칭(Matching) | 기부자가 직접 기관을 선택하거나, 자동 매칭(관리자가   |
|              | 매칭)을 통해 이뤄진다.                                |
+--------------+-------------------------------------------------------+
| 알림         | 회원가입 성공, 기부 등록 및 신청 성공, 기부 물품 상태 |
|              | 변경, 취소 등 특정 이벤트를 기반으로 한 서비스 내     |
| (N           | 알림을 말한다.                                        |
| otification) |                                                       |
+--------------+-------------------------------------------------------+
| 게           | 일반 회원, 기관, 관리자가 이용 가능한 게시판을        |
| 시판(Board)  | 말한다. 게시판을 통해 기관이 물품을 요청할 수 있고,   |
|              | 일반 회원과 기관들이 기부하거나 기부 받은 물품에 대한 |
|              | 후기를 작성할 수 있다. 관리자는 해당 게시판에 작성된  |
|              | 게시글들을 삭제할 수 있으며 문제가 되는 게시글이 있을 |
|              | 시 게시글을 작성한 회원에 대해 조치할 수 있다.        |
+--------------+-------------------------------------------------------+
| 토큰(Token)  | 이메일 주소와 매핑되어, 인증을 진행하는데 있어 고유   |
|              | 식별자로 사용된다.                                    |
|              |                                                       |
|              | 즉, 인증 전용 임시 키값으로 쓰인다.                   |
+--------------+-------------------------------------------------------+
